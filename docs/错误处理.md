# 错误处理

两个维度解耦：控制信息（该不该继续）和诊断信息（出问题看什么）。单用返回值丢详情，逐层打日志拆上下文，全局状态多线程互踩。

方案概览

仿 try-catch，但 error context 是 thread_local 的隐式栈，由三个 API 驱动：

┌─────────────────────────┬────────────────────────────────────┬──────────────────┐
│           API           │                作用                │      调用者      │
├─────────────────────────┼────────────────────────────────────┼──────────────────┤
│ ex_try                 │ 重置上下文，开启新一轮错误记录     │ 关心错误的父函数 │
├─────────────────────────┼────────────────────────────────────┼──────────────────┤
│ ex_hook(errno, "位置") │ 往上下文追加诊断信息               │ 出错的子函数     │
├─────────────────────────┼────────────────────────────────────┼──────────────────┤
│ ex_catch / ex_end     │ 判断是否有错 / 取走完整诊断链+重置 │ 父函数 catch 块  │
└─────────────────────────┴────────────────────────────────────┴──────────────────┘

调用约定（由返回类型决定）

- void 函数：不关心它的返回值（没有返回值可关心）。内部自己 try-catch-end 消化，错误内部闭环，不污染上层
- int 函数：返回错误码。只 hook，不开 try，由调用者 catch-end

签名即契约，不需要文档。

分层职责

父函数   try → 调子函数 → catch → end → 记日志/跳转控制流
子函数 检查下一层返回值 → hook("我在做什么")

- 控制流：靠返回值，只管该不该继续
- 诊断流：靠 hook/catch，在一个问题上下文内累积，只管出问题时能看到什么

嵌套

由约定保证不冲突：父开了 try 表明关心子函数，子函数就不该自己再 try。void 子函数自开 try 属于子函数作用域内的操作，不影响父。

线程安全

错误上下文 是 thread_local 的，各线程独立。

核心设计决策

不逐层打日志：一个错误的完整调用路径是一条日志，不是多条分散日志
try 即重置，end 即消费，不需要复杂的生命周期管理
