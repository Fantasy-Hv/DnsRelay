# 一、错误传递与处理

让错误传递链的行为完全可预测，不给调用者埋坑。

一个函数，要么处理错误，要么传播错误，别两头都占。

传播错误：返回或向上抛出完整、包裹好的错误（不写日志、不吞掉）。

处理错误：捕获它，执行降级、重试、记日志等动作，然后不再向上扔，或者抛出一个全新的业务级别错误。

禁止：捕获、记一条 ERROR 日志、然后又原样抛出去——这会导致日志刷屏，且根源被掩盖。

错误链原则：永不丢弃原始错误。

每次向上包装，必须包裹原始错误，形成一条“cause → cause → root cause”的链。

可靠实践：failed to load config: file not found: /etc/app.yaml (system error: permission denied)。最终日志里缺少任何一环都算不合格。

# 二、日志级别

    （INFO）
    记录“发生了什么业务大事”。例如：用户订单创建、支付成功、服务启动完成。
    特征：每条都是一个里程碑，可被非技术人员理解。

    异常/错误日志（ERROR/WARN）
    记录“什么功能没干成”。例如：发送消息超时（已重试3次失败）、磁盘空间不足。
    特征：需要通知人，但必须包含明确的对象、影响范围、处理建议。

    开发诊断日志（DEBUG/TRACE）
    记录“程序的微观行为”。例如：函数入参出参、内部循环计数器。
    特征：默认关闭，问题排查时动态打开。

铁律：每一条日志都必须能在 1 秒内决定属于哪个抽屉。如果某个错误即属于 B（需要告警）又属于 C（仅调试用），说明你的错误处理层次混乱。
