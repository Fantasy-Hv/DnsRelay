# 项目需求

根据您提供的《计算机网络》课程设计PPT，以下是"DNS中继服务器"项目需求的全面细致总结：

---

## 一、核心功能需求

### 1. 三级查询响应机制

程序需读入"IP地址-域名"对照表，对客户端查询进行三种处理：

| 检索结果              | 处理方式                      | 功能定位           |
| ----------------- | ------------------------- | -------------- |
| IP地址为 **0.0.0.0** | 向客户端返回"域名不存在"的报错消息        | **不良网站拦截功能**   |
| 普通IP地址            | 向客户端返回该地址                 | **本地DNS服务器功能** |
| 表中未检索到该域名         | 向因特网DNS服务器发出查询，并将结果返回给客户端 | **中继功能**       |

### 2. 多客户端并发支持

- 必须支持**多个计算机上的客户端同时查询**
- 允许第一个查询尚未得到答案前就启动处理另外一个客户端查询请求
- 核心机制：**消息ID的转换**（DNS协议头中ID字段的作用）

### 3. 超时处理机制

- 由于UDP的不可靠性，需处理使用外部DNS服务器（中继）时**不能得到应答**或**收到迟到应答**的情形
- 需要设计可测试的超时处理方案

## 二、运行环境与约束

### 1. 技术栈要求

- **操作系统**：Windows 或 Ubuntu（等Linux发行版）
- **编程语言**：C语言
- **网络编程**：Socket编程（UDP协议）
- **DNS协议标准**：RFC 1035（DOMAIN NAMES - IMPLEMENTATION AND SPECIFICATION）

### 2. 字节序处理

- 必须正确处理CPU字节顺序（Big Endian / Little Endian）与**网络字节顺序**的转换
- 使用库函数：`htonl`/`ntohl`（四字节整数）、`htons`/`ntohs`（两字节整数）

### 3. Windows特定要求（如选Windows平台）

- 需链接 `wsock32.lib` 库
- UDP收发使用 `recvfrom` / `sendto` 函数

## 三、DNS协议实现细节

### 1. 报文结构处理

DNS报文由5部分构成，必须完整解析和构造：

| 部分             | 说明         | 长度/特点  |
| -------------- | ---------- | ------ |
| **Header**     | 固定长度报头     | 12字节   |
| **Question**   | 查询问题       | 变长     |
| **Answer**     | 回答的资源记录    | 0~n个RR |
| **Authority**  | 指向权威服务器的RR | 0~n个RR |
| **Additional** | 附加信息RR     | 0~n个RR |

### 2. Header字段必须正确处理

- **ID**：16位标识符，由客户端设置，服务器返回，用于匹配请求与响应
- **QR**：0=查询报，1=响应报
- **OPCODE**：通常为0（标准查询），支持1（反向查询）、2（服务器状态请求）
- **AA**：权威答案标志
- **TC**：截断标志（应答超512字节时置1）
- **RD**：期望递归（查询中设置，告诉服务器处理递归查询）
- **RA**：递归可用（响应中返回，表示服务器支持递归）
- **Z**：保留字段，必须为0
- **RCODE**：响应码（0=无差错，3=名字差错/域名不存在）
- **QDCOUNT/ANCOUNT/NSCOUNT/ARCOUNT**：各部分的记录数量

### 3. Question Section格式

- **QNAME**：域名（如 `www.bupt.edu.cn`）
- **QTYPE**：查询类型（A=1, MX=15, CNAME=5, PTR=12等）
- **QCLASS**：查询类（IN=1）

### 4. Resource Record格式

- **NAME**：域名
- **TYPE**：RR类型码（A=1, AAAA=28, PTR=12, CNAME=5, HINFO=13, MX=15, NS=2等）
- **CLASS**：通常为IN(1)
- **TTL**：生存时间（秒），决定客户端DNS缓存时长
- **RDLENGTH**：资源数据长度
- **RDATA**：资源数据（如A记录为4字节IP地址）

### 5. 报头解析方式

- **禁止使用逐字节/逐比特解析**
- 必须使用C语言**结构体类型定义**进行解析
- 示例：将接收缓冲区强制转换为 `struct HEADER *` 后直接访问字段

## 四、命令行接口规范

### 参考实现语法

```
dnsrelay [-d | -dd] [dns-server-ipaddr] [filename]
```

| 运行模式                                       | 调试级别              | 外部DNS服务器        | 配置文件                    |
| ------------------------------------------ | ----------------- | --------------- | ----------------------- |
| `dnsrelay`                                 | 无                 | 默认 202.106.0.20 | 默认 `dnsrelay.txt`（当前目录） |
| `dnsrelay -d 192.168.0.1 c:\dns-table.txt` | 级别1（时间坐标、序号、查询域名） | 指定 192.168.0.1  | 指定 `c:\dns-table.txt`   |
| `dnsrelay -dd 202.99.96.68`                | 级别2（冗长调试信息）       | 指定 202.99.96.68 | 默认 `dnsrelay.txt`       |

## 五、测试与验证要求

### 1. Windows系统测试步骤

1. 使用 `ipconfig/all` 记录当前DNS服务器（如 `202.106.0.20`）
2. 将系统DNS设置为 `127.0.0.1`（本地主机）
3. 运行 `dnsrelay` 程序（程序内部将外部DNS设为前面记录的地址）
4. 正常使用 `ping`、`ftp`、IE等验证名字解析工作正常

### 2. 测试命令

| 命令                         | 功能                |
| -------------------------- | ----------------- |
| `nslookup www.bupt.edu.cn` | 向名字服务器查询域名对应的IP地址 |
| `ipconfig/displaydns`      | 查看当前DNS缓存内容       |
| `ipconfig/flushdns`        | 清除DNS缓存中所有记录      |

## 六、代码质量与工程规范

### 1. 格式与可读性

- 代码必须**整洁、美观**
- 空格使用规范：该有的必须有，不该有的不要有
- 注意**缩进**和必要的**换行**
- 避免过多缩进（反映整体设计布局问题）

### 2. 架构设计

- **禁止使用两个socket**（反映对socket机制研究不到位）
- **禁止忙等待**：CPU不得每秒进行成千上万次几乎不会命中的探测
  - 必须检查CPU占用率，实际可能只需0.001%算力，不得占用100%
- **必须划分子程序**：禁止一个 `main` 函数流水到底
  - 整个工程禁止只有一个 `.C` 文件

### 3. 注释规范

- 避免**不必要的注释过多**
- **禁止注释中存在BUG**：错误的注释不如没有

### 4. 命名规范

- 变量和函数取名**不得太随意**
- 名字必须与实际功能一致

### 5. 调试信息封装

- 调试信息输出必须**封装**
- 禁止调试相关代码过度侵入主体功能
- 不需要调试时，调试信息应**几乎不占用CPU**
- 推荐封装成**宏和相关子程序**
